## What Each Gap Means in Practice

**Math.random() vs SHA-256** — This is the most dangerous one. If players ever discover the RNG isn't provably fair, trust collapses instantly. A crash game's credibility depends entirely on verifiable fairness. This alone blocks a real launch.

**Missing betting phase** — Right now players likely can't place bets during the correct window. The 4-phase loop (`waiting → betting → flying → crashed`) is what makes the game playable properly. Without the betting phase, the timing logic is wrong.

**Auto-cashout stored but ignored** — This is a silent money bug. A player sets auto-cashout at 2.0x, the rocket passes 2.0x, it never fires, the rocket crashes, the player loses money they should have won. That's a trust-destroying bug the moment real money is involved.

**No `SELECT FOR UPDATE`** — With concurrent players placing bets, the inline SQL can double-spend wallets. Two requests hit the wallet at the same time, both read the same balance, both deduct, one deduction is lost. With M-Pesa real money this is a financial integrity failure.

**No bet limits** — Someone could programmatically bet 0 or bet 999,999,999. No guard means you can be gamed.

**No crash recovery** — If the server restarts mid-game (deployment, crash, OOM kill), all open bets are frozen forever with no refund. Players lose money.
